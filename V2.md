# V2: Desktop-Mobile Sync & Remote Control

## Context

Use the **desktop app as primary** and **mobile phone as a remote control** to continue/control what's on the desktop. Inspired by [Happy](https://github.com/slopus/happy) — a mobile remote for Claude Code with E2E encryption, QR pairing, and WebSocket sync.

---

## Current State

**What already exists:**
- Tauri 2 desktop app with embedded **HTTP server (Axum)** already running
- REST API endpoints: `GET/POST/PATCH/DELETE /api/tasks`, `/api/session`, `/api/settings`, `/api/reports`, `/api/prompt/improve (SSE)`, `/api/prompt/run (SSE)`
- QR code display on `RemoteAccessPage` showing local URL
- Local IP detection + dynamic port
- SQLite as single source of truth on desktop
- Zustand stores with `syncStore.ts` for state coordination
- **Mobile mode** already built into 14 components with `isWebBrowser()` detection and auto-enable

**What's missing:**
- No authentication on HTTP API (anyone on WiFi can hit it)
- No device pairing/linking mechanism
- No real-time push (mobile must poll or use SSE)
- No transport abstraction (IPC calls won't work in browser)
- Same-WiFi only (no internet relay)

---

## Feasibility

### Verdict: **MEDIUM difficulty (4-5/10) — NO major refactor needed**

The architecture is already 60% there. The embedded HTTP server + REST API is the hardest part and it's done. What remains is layering on top.

### What Happy Does vs What We Need

| Feature | Happy (Full) | What We Need (Simpler) |
|---------|-------------|------------------------|
| E2E encryption | AES-256-GCM + NaCl | API token auth (sufficient for LAN) |
| Device pairing | QR → public key exchange → server | QR → shared secret/token (simpler) |
| Sync protocol | WebSocket + versioned messages | SSE (already have) + polling |
| Mobile app | React Native (Expo) | Reuse existing React app in browser |
| Server | Separate PostgreSQL server | Desktop IS the server (already done) |
| Multi-device | Cloud relay | Cloudflare Tunnel for public access |

**Key insight:** Happy is overkill for this use case. Happy needs a cloud server because it relays between CLI sessions across the internet. Our desktop IS the server — mobile just talks to it directly.

## Implementation Plan

### Phase 1: Secure the HTTP API (~1-2 days)
- Generate a random API token on first launch, store in SQLite
- Add token middleware to Axum HTTP server (`Authorization: Bearer <token>`)
- Encode token into QR code URL (e.g., `http://192.168.1.5:7734?token=abc123`)
- Mobile scans QR → gets URL + token → authorized
- **Files:** `src-tauri/src/main.rs` (or wherever HTTP server is defined), settings table

### Phase 2: Real-Time Push via SSE (~1 day)
- Add SSE endpoint `GET /api/events` that streams state changes
- When desktop Zustand store changes → broadcast via SSE channel
- Mobile subscribes on connect, gets real-time updates
- **Files:** HTTP server handlers, add Tokio broadcast channel

### Phase 3: Reuse Existing App as Mobile Web UI (~1-2 days)

**Key discovery:** The existing React app already has full mobile mode support:
- `mobileStore.ts` has `isWebBrowser()` that detects non-Tauri environments
- Auto-enables mobile mode when accessed from a browser (line 22-24)
- 14 components already adapt to `mobileMode` (Sidebar, TopBar, TaskItem, TaskForm, etc.)

**What to do:**
1. **Transport abstraction in `src/lib/tauri.ts`** — the single file where all `invoke()` calls live
   - If `isWebBrowser()` → use `fetch()` against HTTP API (e.g., `getTasks(date)` → `GET /api/tasks?date=...`)
   - If Tauri → use `invoke()` (unchanged)
   - This is ~50 lines of conditional logic in one file
2. **Serve the built app from Axum** — add a static file handler serving `dist/` contents
   - Mobile browser hits `http://host:port/` → gets the React app → auto-enters mobile mode
3. **Handle SSE streaming** — `improvePromptWithClaude` and `runPrompt` already use SSE endpoints, wire them up in browser mode
4. **No new UI code needed** — the existing mobile-responsive components work as-is

**Files to modify:**
- `src/lib/tauri.ts` — add HTTP fetch fallback (main change)
- `src/stores/mobileStore.ts` — minor: ensure browser mode skips Tauri window resize calls (already handled by try/catch)
- Axum HTTP server — add static file serving for `dist/`

### Phase 4: Device Linking UX (~1 day)
- Enhanced `RemoteAccessPage`: show paired devices, revoke access
- Device name/ID stored in SQLite on pairing
- Token refresh/rotation capability
- **Files:** `src/pages/RemoteAccessPage.tsx`, new device management table

### Phase 5: Public Internet Access via Cloudflare Tunnel (~1-2 days)

#### Tunnel Setup
- Tauri app manages `cloudflared` as a child process
- Creates a secure tunnel: `https://planner.yourdomain.com` → `localhost:7734`
- TLS handled automatically by Cloudflare, zero port forwarding
- Mobile accesses the public URL from any network

#### Tunnel Process Management in Tauri
- **Enable/Disable toggle** in Settings UI (persisted in SQLite)
  - Toggle ON → spawns `cloudflared tunnel run` as a child process
  - Toggle OFF → sends SIGTERM to child process, tunnel stops
  - Visual indicator: green/red dot showing tunnel status
- **Settings fields:** Cloudflare tunnel token, enable/disable flag, public hostname display
- **Files:** New Rust module `src-tauri/src/tunnel.rs` + settings UI component

#### Sleep/Wake Auto-Reconnect
- `cloudflared` has **built-in auto-reconnect** with exponential backoff — when laptop wakes from sleep, the tunnel re-establishes automatically (typically 2-5 seconds)
- **Additional safeguard:** Tauri listens for OS wake events via `tauri-plugin-os` or system event hooks:
  1. On wake → check if `cloudflared` process is still alive
  2. If dead → respawn it automatically
  3. If alive → `cloudflared` handles reconnect itself
- **Mobile side:** The mobile web UI shows a connection status indicator. During sleep/offline it shows "Desktop offline — waiting for reconnect..." and auto-retries via SSE reconnect (EventSource auto-reconnects natively)
- **No data loss:** Mobile queues any actions taken while desktop is asleep, replays them on reconnect

#### Security for Public Access
- Phase 1 Bearer token becomes critical (not optional on public internet)
- Add rate limiting to Axum (`tower-governor` crate)
- Consider short-lived tokens with refresh, or TOTP-based pairing
- Optional: Cloudflare Access for email/OTP as a second auth gate
- Audit all endpoints to ensure none are unauthenticated

#### Mobile Web UI Adjustability
- The mobile web UI is the **same React app** served by Axum — reuses all existing components
- You can iterate on mobile-specific layouts by adjusting `mobileMode` conditionals in components
- Changes take effect on next build, or with hot reload during dev
- No coupling to tunnel configuration — works on LAN and public URL identically

## What Does NOT Need Refactoring

- **Tauri IPC layer** — stays as-is, desktop app unchanged
- **SQLite database** — desktop remains single source of truth
- **Zustand stores** — no changes needed, HTTP API reads from same SQLite
- **Existing HTTP API** — just add auth middleware on top
- **Desktop UI** — untouched, mobile reuses the same codebase in browser mode

## Summary

**No big refactor needed.** The architecture is already set up for this — the embedded HTTP server with REST API is the foundation, and the existing mobile mode in the React app eliminates the need for a separate mobile UI. The main work is: (1) add auth to existing API, (2) add SSE for real-time push, (3) transport abstraction in `tauri.ts`, (4) device management, (5) public access via Cloudflare Tunnel.

**Estimated total: ~5-7 days of focused work** (Phase 1: ~1-2d, Phase 2: ~1d, Phase 3: ~1-2d, Phase 4: ~1d, Phase 5: ~1-2d).
